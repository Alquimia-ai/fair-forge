# PromptEvaluator Dataset ‚Äî Design Spec

**Date:** 2026-03-26
**Branch:** `feat/prompt-evaluator`
**Goal:** Create a dataset that demonstrates the value and variability of the PromptEvaluator metrics (CSR, Stability, RSS, ICR, JQ), integrated into fair-forge and referenced in the academic paper.

---

## Context

The `PromptEvaluator` metric and its academic paper are complete. The missing piece is a dataset that:
1. Serves as **synthetic fixtures** for the test suite (deterministic, in-repo)
2. Serves as a **real experiment dataset** that produces meaningful numbers for the paper

The paper proposes 5 signals with distinct root causes. The dataset must exercise each signal with enough variability to demonstrate diagnostic value.

---

## Dataset Structure

### Domains

Three domains, each designed to exercise different signals:

| Domain | Primary Signals | Description |
|--------|----------------|-------------|
| **FAQ / Support** | CSR, Stability, RSS | Well-defined questions with a single expected answer. Ground truth present ‚Üí activates RSS automatically |
| **RAG / Knowledge Base** | CSR, Stability, RSS, JQ | Variable context per query. RSS detects hallucinations. JQ evaluates quality when RSS is insufficient |
| **Structured Instructions** | CSR, ICR | Prompts with verifiable constraints (JSON output, word count, keyword inclusion). ICR measures deterministic compliance |

### Prompts per Domain

Each domain has **3 system prompts** designed to produce distinct metric ranges:

| Prompt Type | Expected CSR | Expected RSS/ICR | Purpose |
|-------------|-------------|-----------------|---------|
| **Good** | High (0.85‚Äì1.0) | High (0.80‚Äì1.0) | Establishes the ceiling |
| **Ambiguous** | Medium (0.50‚Äì0.75) | Medium (0.40‚Äì0.70) | Shows diagnostic middle ground |
| **Bad** | Low (0.20‚Äì0.50) | Low (0.10‚Äì0.40) | Demonstrates failure modes |

### Scale

- 3 domains √ó 3 prompts √ó 10 queries = **90 (prompt, query) pairs**
- K=10 samples per query = **900 LLM calls** for the real experiments
- Model: `gpt-oss-20b` via Groq API

## Synthetic Fixtures

Location: `tests/fixtures/` (following existing patterns)

Purpose: deterministic, mock-based fixtures for unit tests. No real LLM calls.

Contents:
- `create_prompt_evaluator_dataset()` ‚Äî returns a `list[Dataset]` covering all 3 domains
- `PromptEvaluatorDatasetRetriever` ‚Äî retriever class wrapping the fixture
- Fixtures in `tests/conftest.py`: `prompt_evaluator_dataset` and `prompt_evaluator_dataset_retriever`

Approach: TDD ‚Äî tests written first, fixtures extracted from test needs.

---

## Real Experiment Dataset

Location: `datasets/prompt_evaluator/` (new directory, committed to repo)

Format: JSON files, one per domain, loadable via a `Retriever` subclass.

Structure per entry:
```json
{
  "session_id": "faq_good_prompt",
  "assistant_id": "gpt-oss-20b",
  "domain": "faq",
  "prompt_type": "good",
  "language": "english",
  "context": "...",
  "conversation": [
    {
      "qa_id": "faq_001",
      "query": "...",
      "assistant": "",
      "ground_truth_assistant": "..."
    }
  ]
}
```

The `assistant` field is left empty ‚Äî PromptEvaluator generates responses itself.

## Visualizations

Generated by `notebooks/prompt_evaluator_experiments.ipynb`.

| Plot | What it shows | Destination |
|------|--------------|-------------|
| **Clustering plot** (t-SNE/UMAP 2D) | K response embeddings colored by semantic cluster ‚Äî visual proof of CSR and SE | Paper + docs |
| **Radar chart** | All 5 signals (CSR, Stability, RSS, ICR, JQ) for the 3 prompt types per domain | Paper |
| **Heatmap query √ó prompt** | CSR or Stability per query for the 3 prompts ‚Äî shows intra-prompt variability | Paper |
| **Bar chart** | Side-by-side comparison of the 3 prompts per domain across all available signals | Paper + README |
| **Scatter CSR vs RSS** | Detects the "consistently wrong" pattern (high CSR, low RSS) | Paper |

All plots exported as PNG to `assets/figures/` for LaTeX inclusion.
